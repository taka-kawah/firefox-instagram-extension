# issue レビュー skill

`taka-kawah/firefox-instagram-extension` の issue を、**リポジトリの設計と既存コードに照らして**レビューする。
単なる感想ではなく、「これは妥当か」「どこを直すことになるか」「実装方針はどうなるか」「何が曖昧か」を具体的に示すのが目的。

## このリポジトリの前提知識（レビューの土台）

- **目的**: Instagram の Web UI から、おすすめ投稿・リール導線・発見ページのおすすめを隠し、フォロー中の投稿に集中できるようにする Firefox 拡張（MV3, ビルド不要の素のJS）。
- **ロードマップ**: `README.md` にフェーズ1〜5の計画がある。issue がどのフェーズに当たるか／既に計画済みでないかを必ず確認する。
- **作業規約**: `AGENTS.md`（PR本文は初心者にも分かるよう丁寧に・テスト手順必須 等）。
- **主要ファイルと役割**:
  - `manifest.json` … 権限・content_scripts の設定
  - `src/content/content.js` … メイン処理（おすすめ投稿の判定・MutationObserver での追従・設定反映）
  - `src/content/labels.js` … 「おすすめ」多言語ラベル辞書（判定の根拠）
  - `src/styles/hide.css` … リール導線・発見グリッドを属性フラグで隠すCSS
  - `src/popup/*` … オン/オフ設定パネル
  - `src/background/background.js` … 設定初期値の保存
- **設計上の使い分け**: 「常に同じ場所を隠す」→ CSS（`html` の `data-iff-*` フラグ＋`hide.css`）。「中身を見て判断」→ JSで1件ずつ判定。
- **既知の壊れやすさ**: Instagram の HTML はクラス名が自動生成で頻繁に変わる → セレクタは `href` や表示テキストなど壊れにくい手がかりを使う。ラベルは表示言語依存。

## 手順

### 1. issue を取得
`mcp__github__issue_read`（method=get）で本文・ラベル・コメントを取得する。関連issueやロードマップとの重複も確認（必要なら `mcp__github__list_issues`）。

### 2. リポジトリと突き合わせ
- README のフェーズ計画に該当するか／重複・先取りでないか。
- 関係しそうなファイルを実際に読む（上記の主要ファイル）。**どの既存パターンで実現できるか**を当てる
  （CSSフラグ方式で済むのか、JS判定が要るのか、ラベル辞書の追加で済むのか 等）。

### 3. レビューを構造化してまとめる
以下の見出しで日本語でまとめる（該当しない項目は省略可）:

1. **要約** — このissueが求めていること（1〜3行）
2. **種別 / 対象フェーズ** — bug / 機能要望 / 質問 / 改善、README のどのフェーズか
3. **妥当性・重複** — 目的に合っているか、既存issueやロードマップと重複しないか
4. **影響範囲** — 変更が必要になりそうなファイル（`content.js` / `labels.js` / `hide.css` / `manifest.json` / popup 等）
5. **実装方針案** — このリポジトリの設計に沿った具体的な進め方（CSSフラグ vs JS判定、ラベル追加、新しい `data-iff-*` フラグ追加 など）
6. **リスク・制約** — IGのDOM変化への弱さ、多言語ラベル漏れ、MV3 制約、誤検出（短文ラベルの完全一致）等
7. **受け入れ条件の明確さ / 不明点** — 「何が起きれば完了か」が定義済みか。曖昧なら著者への確認事項を箇条書き
8. **推奨** — ラベル案・優先度・次アクション（必要なら「動作確認は `verify-instagram-extension` skill で」）

### 4. コメント投稿（依頼された場合）
`mcp__github__add_issue_comment`（owner=`taka-kawah`, repo=`firefox-instagram-extension`, issue_number=対象番号）で投稿する。
- 末尾に `🤖 Generated with [Claude Code](https://claude.com/claude-code)` を付ける。
- **投稿は依頼されたときのみ**。まずレビュー結果を提示し、投稿要否を確認するのが基本。

## 完了の定義
- 上記の構造でレビューがまとまり、リポジトリの設計・実コードに基づいた具体的な指摘になっている
  （一般論で終わっていない）。依頼された場合は issue にコメント投稿済み。
